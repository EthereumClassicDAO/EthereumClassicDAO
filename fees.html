ECIP-1111 & ECIP-1115 · Fees & Miner Incentives

Modern Fees, Non-Inflationary Funding, Optional Smoothing

Olympia updates Ethereum Classic’s fee system to use EIP-1559-style BASEFEE accounting, while preserving Proof-of-Work and fixed emission. ECIP-1111 introduces modern transaction types and BASEFEE redirection; ECIP-1115 adds an optional smoothing mechanism for miner incentives.

This page explains how fees work before and after Olympia, what changes for users, wallet providers, miners, and node operators, and how the optional smoothing layer fits into the broader design.

Current ETC Fee Model vs. Olympia

Today, Ethereum Classic uses a legacy gas price model: transactions specify a single gasPrice, miners choose which transactions to include, and the entire fee (gasUsed × gasPrice) goes directly to the miner.

This model works, but it has tradeoffs:

  • Fee estimation can be noisy and imprecise, especially under congestion.
  • Wallets must guess an appropriate gas price, often overpaying to ensure inclusion.
  • There is no protocol-level concept of a BASEFEE, limiting compatibility with modern EVM tooling that expects EIP-1559 semantics.
  • All protocol revenue goes directly to miners; there is no built-in way to fund public goods.

Olympia addresses these issues by:

Adding EIP-1559-style BASEFEE Introducing type-2 transactions Redirecting BASEFEE (instead of burning) Keeping block rewards & PoW unchanged Optionally smoothing fee revenue

ECIP-1111 – Olympia EVM & Protocol Upgrades

ECIP-1111 integrates EIP-1559-style fee mechanics into Ethereum Classic and enables BASEFEE redirection. Conceptually, it introduces two key elements:

  • A per-block BASEFEE that adjusts up or down depending on recent block utilization, making fee estimation more predictable.
  • Type-2 transactions with max_fee_per_gas and max_priority_fee_per_gas, widely used across the EVM ecosystem.

Mechanics How BASEFEE Works

Each block has a BASEFEE value. When blocks are more full than the target gas usage, BASEFEE gradually increases; when blocks are less full, it decreases. This creates a self-adjusting price signal for block space.

For a transaction:

  • The user specifies a max fee and a tip (priority fee).
  • The protocol charges BASEFEE + tip up to the max the user allowed.
  • Under Olympia, BASEFEE is redirected instead of burned.

Compatibility EVM & Tooling Alignment

With ECIP-1111, Ethereum Classic becomes compatible with:

  • Wallets and SDKs that assume EIP-1559 fee fields.
  • Gas estimation libraries built around BASEFEE.
  • Smart contracts that reference BASEFEE via the corresponding opcode (e.g., pricing oracles).

This reduces friction for users and developers moving between ETC and other EVM chains.

Incentives What Changes for Miners?

Miners still receive:

  • The full block reward (unchanged fixed emission).
  • Priority fees (tips) from included transactions.

The difference is that BASEFEE is no longer paid directly to miners. Instead, it is redirected to a consensus-chosen destination—typically the Olympia Treasury when ECIP-1112 is enabled.

This creates a shared, protocol-level funding pool while preserving immediate incentives via tips.

Key Point: ECIP-1111 modernizes ETC’s fee system, improves compatibility, and enables non-inflationary protocol funding by redirecting BASEFEE. It does not change Proof-of-Work, block rewards, or the fixed-emission schedule.

Where Does BASEFEE Go?

Under Olympia, BASEFEE is not burned. Instead, ECIP-1111 defines a mechanism to redirect BASEFEE to a chosen destination. ECIP-1112 specifies that destination as the Olympia Treasury, an immutable, protocol-controlled contract.

At a high level, you can think of the flow as:

  • Users pay fees: BASEFEE + tip.
  • Miners receive tips + block reward.
  • BASEFEE is redirected into the treasury (once ECIP-1112 is active).

This means:

  • Fee pressure still reflects real demand for block space.
  • Miners remain incentivized via block rewards and tips.
  • The network gains a transparent, on-chain funding pool for public goods.
Non-Inflationary: No new ETC is minted. BASEFEE redirection simply changes where an existing revenue stream goes—from miners or a burn address to a protocol treasury governed by the community.

ECIP-1115 – Optional Miner Incentive Smoothing

BASEFEE can be volatile. In periods of high demand, treasury inflows may spike; in quiet periods, they may drop. ECIP-1115 introduces an optional smoothing mechanism that spreads the impact of BASEFEE over time using a modeled “L-curve”.

Goal Stabilizing the Security Budget

The primary goal of ECIP-1115 is to reduce short-term volatility in miner revenue related to protocol fees, while still preserving the long-term funding benefits of BASEFEE redirection.

  • High-traffic periods don’t immediately translate into sudden, one-off rewards.
  • Low-traffic periods can be cushioned by previously accumulated fee revenue.
  • Miners gain more predictable expectations for total compensation over time.

Mechanism L-Curve Smoothing

Conceptually, smoothing works by releasing BASEFEE-derived value gradually across a window of future blocks, instead of instantly in the block where the fee is paid.

The “L-curve” describes how quickly or slowly this release decays over time. The exact parameters are defined in the ECIP-1115 spec and can be tuned based on modeling and community preference.

Scope Optional & Independent

ECIP-1115 is explicitly optional. The network can:

  • Adopt ECIP-1111–1114 without smoothing.
  • Activate ECIP-1115 later, after observing fee behavior in practice.
  • Refine or replace smoothing parameters through future ECIPs if needed.

This keeps the core Olympia architecture flexible while still offering a path to long-term stability in miner incentives.

Optional by Design: ECIP-1115 is a tool, not a requirement. It lets ETC balance responsiveness to real-time demand with stability in miner revenue, without altering fixed emission or consensus.

FAQ – Fees & Incentives

Does Olympia make transactions more expensive?

Not by design. EIP-1559-style BASEFEE improves predictability, not necessarily average cost. Users can still choose how much to tip for faster inclusion, and wallets can use better estimation algorithms based on BASEFEE.

What changes for wallets and dApps?

Wallets and dApps that already support EIP-1559 on other chains can reuse most of that logic on ETC after ECIP-1111. They’ll set max_fee_per_gas and max_priority_fee_per_gas instead of a single gas price. Existing tooling can be adapted rather than rebuilt.

Do miners lose revenue because of BASEFEE redirection?

Miners continue to receive block rewards and priority fees. BASEFEE is diverted into a shared treasury to fund public goods. ECIP-1115 provides a way to smooth overall revenue and help keep the security budget predictable.

Can the smoothing mechanism be adjusted later?

Yes. Because ECIP-1115 is optional and separate from ECIP-1111–1114, the network can revisit its parameters—or even the approach—through future ECIPs as real-world data accumulates.
